# 灵境运行时与多智能体架构

> author:dengwei date:2026-07-18

这份文档说明灵境 1.1 的运行闭环，以及多智能体之间如何分工、通信和管理上下文。

## 一、整体分层

```text
用户 / VS Code 面板
        |
        v
AgentSession（会话和 RunState）
        |
        v
主智能体循环（规划、调用工具、决定下一步）
   |          |             |
   v          v             v
本地工具   只读子智能体   MCP / Git / 后台进程
   |          |
   +-----+----+
         v
确定性验证 -> 独立审查 -> 最终回复
```

- `AgentSession` 保存消息和当前任务状态，负责开始、停止、恢复和压缩。
- 主智能体拥有最终决策权，是唯一可以组织写入和给出最终答案的角色。
- 工具层负责真实读取、补丁、命令、Git、MCP 和代码索引，模型不能直接碰文件系统。
- 子智能体用于并行调查，不用于并行修改同一工作区。
- 验证流水线由程序触发，不依赖模型是否记得运行测试。
- 独立审查器使用新的短上下文，只看原需求和实际修改前后内容。

## 二、主循环如何完成一次编码任务

1. 根据任务复杂度决定是否生成计划，简单问答不创建计划。
2. 将原始需求、引用文件、附件和适用的 `AGENTS.md` 合并成任务消息。
3. 每轮把消息和按任务裁剪后的工具声明发给模型。
4. 模型调用读取、搜索、符号或诊断工具收集证据。
5. 修改时优先调用 `apply_patch`。所有编辑先在内存验证，全部匹配后才原子写入一次。
6. 程序要求模型重新读取修改区域，避免只写不看。
7. 程序从 `package.json` 选择 compile、lint、test，任一失败就把输出送回主循环继续修复。
8. 独立审查器检查需求覆盖和明显回归；不通过时同样返回主循环修复。
9. 文件清单、计划和验证都完成后，主智能体才能输出最终答复。

机械性任务不会把每个函数当成一次独立循环，而是读取 300-500 行完整区块，收集其中 5-30 个修改后一次提交。`ToolEfficiencyTracker` 会观察成功调用：相同参数重复读取会提示使用已有上下文；同一文件连续提交小补丁会提示合并。成功写入会清除该文件的读取统计，允许重新读取相同范围确认结果。

## 三、多智能体怎么协作

### 1. 什么时候创建子智能体

只有复杂、批量、全面审查、重构或架构分析等任务，主智能体才会看到 `delegate_tasks`。最多并行三个子任务，角色目前有：

- `explorer`：定位代码、梳理调用链和项目结构。
- `tester`：查找验证入口、现有测试和潜在失败点。
- `reviewer`：从只读角度检查风险、遗漏和回归。

主智能体应当把互相独立的调查交出去。例如“查认证调用链”和“查现有测试”可以并行；“先改接口再改调用方”有先后依赖，不适合并行。

### 2. 怎么通信

当前采用星形通信：

```text
子智能体 A --摘要--+
子智能体 B --摘要--+--> 主智能体 --> 文件修改与最终结论
子智能体 C --摘要--+
```

子智能体之间不直接发消息，也不共享可变内存。主智能体通过一次 `delegate_tasks` 调用发送子任务；运行时使用默认并发数 2 的受控任务池执行，失败子任务自动重试一次；每个子智能体结束后返回包含状态、尝试次数和摘要的结构化结果，所有结果合并成一个工具结果加入主会话。这样可以减少服务商并发限流，也避免子智能体互相等待或形成无法追踪的对话环。

### 3. 上下文怎么管理

每个子智能体拥有独立消息数组，只包含：

1. 角色和只读边界的系统消息。
2. 主任务原文。
3. 分配给自己的具体子任务。
4. 自己产生的工具调用和工具结果。

主会话以前的聊天、其他子智能体的过程、图片 Base64 和无关工具结果都不会复制过去。子智能体返回后，主会话只保留结构化摘要，不保留完整推理轨迹。工具调用失败会先作为工具结果反馈给子智能体，允许它修正参数或换用其他只读工具；连续失败达到阈值后才终止。这样既降低 Token 消耗，也减少偶发工具错误对调查结论的影响。

### 4. 为什么子智能体不能写文件

多个执行者同时修改同一个工作区会产生竞态：A 读取旧版本，B 先写入，A 再按旧内容覆盖，最终文件就可能混乱。因此 1.1 采用“并行读取、串行决策、主智能体写入”的保守方案。它已经能加快大项目定位，同时保持修改顺序可追踪。

以后如果需要并行编码，应给每个写入型子智能体创建独立 Git Worktree。子智能体在自己的目录和分支修改、测试并提交，主智能体只接收提交哈希或补丁，再按依赖顺序合并。那是下一阶段，不应该让多个子智能体直接共享当前工作区。

## 四、运行状态与恢复

`RunState` 持续记录：原始任务、状态、循环轮次、计划、文件任务、已修改文件、验证记录、子智能体活动和最近错误。状态通过 VS Code `workspaceState` 按工作区保存。

扩展重启后，原来处于 `running` 的任务会恢复为 `paused`。用户点击继续后，运行时沿用原始任务和已有计划，从保存轮次继续；它不会重新规划，也不会把内部验证消息误认为新的用户需求。点击丢弃会清除本次会话和运行状态。

主智能体每 80 轮建立一次内部检查点，保存状态并压缩上下文后自动继续，用户不需要为普通检查点点击“继续任务”。一次开始或继续操作最多自动跨越 4 个阶段，共 320 轮；只有达到这个总成本安全上限，或者连续 32 轮没有新的有效工具进展时，才进入 `paused`。累计轮次持续写入 `RunState`，手动继续会追加一段新的 320 轮自动执行预算。每个子智能体的最多 24 轮属于它自己的消息循环，主循环只把整次 `delegate_tasks` 记为一轮，因此不会把三个子智能体的内部请求次数累加到主智能体预算。

## 五、权限和安全边界

- 文件路径仍受当前工作区边界检查。
- `read-only` 禁止写文件、后台进程和 Git 写操作。
- `ask` 对每次文件写入弹窗确认。
- 后台进程、Git 写操作和 MCP 调用始终需要确认。
- 这些是 VS Code 扩展内的权限控制，不是操作系统沙箱。
- MCP 服务器属于外部进程，应当只配置可信命令，并通过环境变量传递密钥。

## 六、目前的限制

- 子智能体数量最多 3 个，并发数可以配置，但尚未根据 Token 和实时延迟动态调度。
- 子智能体返回结构化状态和摘要，但证据内容与置信度仍由模型生成，尚未做程序级校验。
- 独立审查仍由同一模型服务完成，只是隔离上下文，不等同于不同供应商交叉审查。
- 并行写入型子智能体尚未启用；需要 Worktree、提交依赖和冲突处理后才能安全开放。
- MCP 当前支持 stdio，尚未加入远程 HTTP/SSE 传输和细粒度服务器白名单。
